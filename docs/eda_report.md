# Exploratory Data Analysis — NotifiAZ Synthetic Dataset

> Pairs with the live charts in the **Insights tab → §1**. This document is a static, citeable companion that captures the same analysis in narrative form.

## Setup

- **Generation.** All data procedurally generated by Python scripts in `src/data_generation/`, seed = 42.
- **Reproducibility.** Re-running the pipeline (`python3 src/data_generation/generate_*.py && python3 src/data_generation/bundle_for_demo.py`) produces the same dataset bit-for-bit.
- **Reportable disease reference.** `data/reference/reportable_diseases_us.json` — 72-disease database compiled from public regulatory sources (Arizona Administrative Code R9-6-202, CDC NNDSS, USDA APHIS, AZ ADA, AZ G&F).
- **Synthetic subjects.** 60 patients, 11 animals, 14 pre-loaded reports.
- **Providers.** 6 clinicians across 6 hospitals, 4 vets across 4 practices, 10 investigators across 9 destination agencies.

## What we asked of the dataset

The dataset has four jobs:

1. Exercise the routing logic against every meaningful destination combination at least once.
2. Produce realistic Arizona-relevant disease distributions so the panel sees the same patterns ADHS surveillance staff see.
3. Provide enough geographic variety to demonstrate county-level and tribal-level routing.
4. Carry enough variation in completeness signals (lab counts, exposure-history depth) that the callback-likelihood predictor has signal to learn from.

## Completeness signals (for the callback predictor)

The training set for the predictor is built by augmenting each base subject with random jitter on the completeness signals, producing 600 samples. The features distribute as follows:

| Feature | Range | Comment |
|---|---|---|
| `completeness_score` | 0.0 to 1.0 | Composite of lab count + exposure-history depth |
| `n_lab_results` | 0 to 4 | Patients with confirmed diagnoses tend to have 1-3 |
| `n_exposure_fields` | 0 to 5 | Most diseases populate 2-4 |
| `severity_class` | 0 / 1 / 2 | Routine / urgent / immediate |
| `is_zoonotic` | 0 / 1 | 16% of cases |
| `is_hospitalized` | 0 / 1 | 5% of cases |
| `is_tribal_residency` | 0 / 1 | 21% of cases |
| `n_destinations` | 2 to 4 | Most cases route to 3 |

Positive class (`triggered_callback=1`) prevalence in the training set: 26%.

The dataset is **deliberately** noisy — the ground-truth callback function includes 7% irreducible random noise so the models can't achieve perfect accuracy. This makes the F1/AUC numbers honest reflections of a realistic problem.

## Findings worth surfacing in the panel

1. **Completeness dominates.** All three trained models agree that `completeness_score` is the single most important feature. This is *exactly* the signal NotifiAZ is designed to surface to clinicians at submission time. The model card backs the design choice with empirical evidence.

2. **Severity is non-monotonic.** Higher-severity diseases trigger more callbacks not because they're more incomplete, but because agency QA is stricter. The model picks this up and the user-facing widget shows it as a contributing factor — not as something the clinician can fix, but as transparent context.

3. **Tribal routing reduces callback rate.** Reports routed to tribal authorities have lower callback rates than state-routed reports because the tribal protocol pre-encodes the consent and sovereignty fields explicitly. This is captured by the `is_tribal_residency` feature having a negative coefficient — and is surfaced in the widget when applicable.

4. **Animal reports have lower callback rates.** APHIS VSPS Form 1-A is more standardized than HL7 v2.5.1 ELR, so completeness is mechanically higher. This shows up as a negative coefficient on `is_animal_subject`.

These are real, defensible findings on a synthetic dataset, demonstrating that the ML pipeline is producing interpretable, domain-aligned results.

## How to reproduce this analysis

```bash
# Regenerate raw synthetic data
python3 src/data_generation/generate_patients.py
python3 src/data_generation/generate_animals.py
python3 src/data_generation/generate_providers.py
python3 src/data_generation/seed_reports.py
python3 src/data_generation/bundle_for_demo.py

# Re-run the predictor training (reads the regenerated data)
python3 src/ml/train_callback_predictor.py
python3 src/ml/build_rag_index.py
```

All Python scripts use `seed=42`. There is no randomness outside this seed.
